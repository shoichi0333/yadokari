# YADOKARI UI/UX レビュー & 次回対応リスト

作成日: 2026-05-15  
対象バージョン: Phase 1（静的データ・モック物件・6都道府県届出データ）

---

## 1. 使いやすい点（残すべき）

- **可否チェッカー（/check）**: 住所入力 → 即判定の流れは明快。人気エリアのショートカットも有効
- **セグメントカード（トップ）**: 4ターゲット別の導線が視覚的に分かりやすい
- **民泊タイプ解説セクション**: JUUTAKU/TOKKU/RYOKAN の違いを比較表形式で整理、初心者に親切
- **競合マップのクラスタリング**: 2,441件を固まりで表示、パフォーマンス的に良い
- **FAQ JSON-LD（/check）**: SEO構造化データが正しく実装されている
- **PropertyCard のタグチップ**: エメラルドカラーで視認性が高い

## 2. 使いにくい点・改善が必要なもの

### A. ナビゲーション / 情報設計

| 問題 | 場所 | 深刻度 |
|---|---|---|
| `/map` と `/listings` が同一URL（リダイレクト）なのにヘッダーに両方リンクがある | Header.tsx | ★★★ |
| 「競合マップ」と「エリアを探す」の違いが名前から分からない（/map と /search） | Header, BottomNav | ★★★ |
| デスクトップナビに「民泊コラム（/blog）」がない（モバイルとフッターにはある） | Header.tsx | ★★ |
| ボトムナビに「料金プラン」がない → モバイルでの有料化導線がゼロ | BottomNav.tsx | ★★ |
| ハンバーガーメニューとボトムナビが両方表示される（モバイル） → 重複・迷い | Header + BottomNav | ★★ |

**対応案:**
- `/listings` → 削除して `/map` に統一。ヘッダーリンクも1本に
- ヘッダーの「競合マップ」→「届出住宅マップ」に改名して差別化を明確化
- デスクトップナビに「コラム」追加
- ボトムナビに5項目目「料金」を追加（またはダッシュボードと入れ替え）

### B. コピーライティング / メッセージの矛盾

| 問題 | 場所 | 深刻度 |
|---|---|---|
| ヒーローコピー「民泊できるエリアを、正確に探す」→ 実際は物件検索サイトではなく「エリア分析 + 外部誘導」ツール | page.tsx | ★★★ |
| SearchFormのplaceholder「エリア・駅名・物件名で検索」→ /search はエリア検索。「物件名」は検索できない | SearchForm.tsx | ★★ |
| 使い方Step 03「物件を外部サイトで探す」→ 自サービスで完結しない終点を公言している | page.tsx | ★★ |
| 物件詳細パンくず「物件検索」→ クリックすると /search（エリア検索）に飛ぶ | property/[id]/page.tsx | ★★ |

**対応案:**
- ヒーローコピー → 「民泊の可否を、住所1つで調べる。」など実態に即したコピーへ
- SearchForm placeholder → 「エリア・駅名で検索」（物件名を除外）

- ヘッダーtagline → 「エリア分析・可否判定・収益試算」など複数機能を示す表現へ

### C. UIの不整合・バグ級の問題

| 問題 | 場所 | 深刻度 |
|---|---|---|
| 競合マップの `SOURCE_NAMES` が2件のみ（minato/nakano）で古い → 6ソース分のエリア名が生ID表示になる | app/map/page.tsx:43-46 | ★★★ |
| `SOURCE_NAMES` に中野区が定義されているが実データに中野区は存在しない | app/map/page.tsx:44 | ★★ |
| 料金プランのStripe環境変数未設定時の状態が不明 → ボタンが壊れて見える可能性 | app/pricing/ | ★★ |
| TYPE_FILTERS に英語コードが混在「住宅宿泊 (JUUTAKU)」→ 一般ユーザーには不要な表記 | SearchResultsClient.tsx:24-27 | ★ |
| 競合マップページが独自 `<header>` を持つ → layout.tsx の `<Header>` と2重になりうる | app/map/page.tsx:130 | ★★ |

- 地図ページの独自headerを削除してレイアウトの共通Headerを利用する
- サンプルバナーを小さく目立たないデザインへ、もしくは削除してフッターに注釈

### D. セクション順序の不自然さ（トップページ）

現在の順序：
```
ヒーロー → 統計バナー → 特徴（選ばれる理由）→ セグメントカード → 使い方3ステップ → 民泊タイプ解説 → ブログ → 競合マップCTA
```

望ましい順序：
```
ヒーロー → 統計バナー → セグメントカード（誰向け？）→ 特徴（なぜ使う？）→ 使い方3ステップ → 民泊タイプ解説 → ブログ → CTA
```

ユーザーは「自分向けか確認」してから「なぜ良いか納得」する順が自然。現在は逆。

### E. 機能の未完成 / 不整合

| 問題 | 場所 | 深刻度 |
|---|---|---|
| 今回追加したタグフィルタにリセット手段がない | SearchForm.tsx | ★★ |
| SearchFormのタグがURLパラメータに乗るが、/search（エリア検索）はtagsを解釈しない → フィルタが効かない | SearchForm + SearchResultsClient | ★★★ |
| お気に入りボタン（FavoriteButton）が未ログイン時にどう振る舞うか不明 | FavoriteButton.tsx | ★ |
| エリアページ（/area/[prefecture]）の対応都道府県がWARD_ZONING_MAP依存で限定的 | area/[prefecture] | ★ |

**特に重要（★★★）:**  
SearchFormで「ペット可」を選択して検索すると `/search?tags=ペット可` になるが、`/search`（エリア検索ページ）はタグを読まない。タグフィルタは実際には物件一覧（`/property`系）に対して機能すべきだが、その物件一覧ページが現状存在しない。タグフィルタの流入先を整理する必要がある。

## 3. デザイン評価

### 良い点
- teal / emerald のカラーパレットは一貫していて民泊（清潔・自然）イメージに合う
- カード系UIはrounded-2xl + shadow-smで統一されており品質感がある
- MinpakuBadgeのカラーコード（緑/金/紫/赤）が直感的

### 改善点
- **フォントサイズが全体的に小さい** → 説明テキストが text-sm / text-xs が多く、特にモバイルで読みにくい
- **ヒーローのグラデーション**（teal→emerald）は綺麗だが、テキストの可読性が下がる箇所がある（teal-100テキスト on teal-600背景）
- **セグメントカードのhover効果なし** → 今回追加したカードはhoverがなくインタラクティブ感が薄い。shadow-md + border-teal-200 on hoverを追加すべき
- **ブログカードに画像がない** → minpaku-information.com には画像付き記事があり視覚的に負けている
