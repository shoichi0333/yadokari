# YADOKARI 成長・UX改善メモ

作成日: 2026-05-17

## 結論

YADOKARI が儲かる形に近づくには、単なる「物件検索」よりも、民泊を始めたい人の不安を順番に潰す「判断支援サービス」に寄せるのがよい。

ユーザーが本当に知りたいのは、次の4つ。

1. この住所で民泊できるか
2. どの制度で運営すべきか
3. 競合が多すぎないか
4. 結局いくら儲かりそうか

この4つを無料でざっくり見せ、詳細なレポート・通知・掲載・問い合わせ管理を有料化する。

## 天才エンジニア目線の改善

### 追加したほうがいい

- 判定結果の `score` 化
  - 例: 民泊ポテンシャル 82/100
  - 初心者でも一瞬で判断できる
  - SEO記事や広告LPでも訴求しやすい

- 判定結果ごとの next action
  - 可なら「管理規約確認」「消防設備」「収益詳細」
  - 不可なら「近隣の商業系エリアを探す」
  - 今日 `/check` の結果画面に追加済み

- DB側の server-side plan enforcement
  - localStorage の plan は便利だが、本番では不正利用を防げない
  - `/api/listings`, `/api/favorites`, 詳細レポート API で必ず DB plan を見る

- Contact/Lead の永続保存
  - 現在 fallback log まで
  - 次は `Lead` / `ContactInquiry` model を Prisma に追加し、メール未設定でも確実にDB保存

- 競合データ schema validation
  - `minpaku_listings.json` が壊れてもUIは落ちないようにした
  - 次は scraper 出力時点で zod などで検証したい

### いらない、弱い

- 「物件検索サイトです」と強く言い切ること
  - 現状はリアルタイム賃貸在庫サイトではない
  - 「民泊判断支援」「候補探索」「外部物件確認の前段」に寄せたほうが信頼を損ねにくい

- コラムを主要モバイルナビに置くこと
  - SEOには必要だが、収益導線としては弱い
  - モバイル下部ナビは「料金」に変更済み

- ロゴ画像の多用
  - 小さい画像ロゴは白背景が出て安っぽく見えた
  - ヘッダー/フッターはテキストワードマークへ寄せた

## 民泊を始めたいユーザー視点の改善

### 追加したほうがいい

- 「この物件で最初に確認すべきこと」チェックリスト
  - 管理規約
  - オーナー承諾
  - 消防設備
  - 近隣説明
  - 鍵受け渡し
  - 清掃導線

- 「民泊不可だった場合の代替案」
  - 近隣エリアを探す
  - 旅館業向け商業地域に絞る
  - 特区エリアを見る

- 収益の見せ方
  - 月商だけでなく、清掃費・手数料・家賃を引いた利益を強調する
  - 初心者は売上と利益を混同しやすい

- レポート化
  - 住所ごとのPDF/メールレポート
  - 無料は簡易、有料は詳細
  - これは課金理由としてかなり強い

## UI・デザイン方針

- SaaS/不動産ツールとして、派手な装飾より「判断しやすい密度」を優先する
- 角丸は少し締める。カードは `rounded-xl` 程度が信頼感に合う
- CTAは黒または濃いtealで明確にする
- 操作できるカードはカード全体クリックにする
- モバイルでは「可否チェック」「物件」「マップ」「料金」を優先する

## 儲かりそうな機能

### 1. 詳細エリアレポート

無料:

- 可否
- 競合件数
- 簡易収益

有料:

- 競合密度比較
- 制度別おすすめ運営形態
- 清掃費・手数料込み利益
- チェックリスト
- 外部物件リンク
- メール通知

### 2. 物件URL貼り付け診断

SUUMO/アットホームURLを貼るだけで、住所・賃料・間取りから診断する。

広告訴求しやすい:

> 「気になる物件URLを貼るだけで、民泊向きか10秒診断」

### 3. 不動産会社向け掲載

Pro:

- 民泊向け物件掲載
- 問い合わせ管理
- エリアレポート添付
- 反響分析

### 4. 監視・通知

Standard:

- お気に入りエリアの新規届出通知
- 候補物件の類似エリア通知
- 競合増加アラート
