# YADOKARI 要件定義・ロードマップ

作成日: 2026-05-17

## 1. プロダクトの目的

YADOKARI は、民泊を始めたい個人・投資家・不動産事業者が、候補住所や物件について次の判断を短時間で行えるサービスにする。

- その場所で民泊運営ができる可能性があるか
- どの制度区分で検討すべきか
- 周辺にどれくらい届出住宅・競合があるか
- 賃料に対して収益が見込めるか
- 気になる物件・エリアを継続的に追えるか

## 2. 対象ユーザー

| ユーザー | 主な課題 | 成功条件 |
| --- | --- | --- |
| 民泊初心者 | 住所や用途地域を見ても可否が分からない | 住所入力だけで制度区分と注意点が分かる |
| 民泊投資家 | 物件候補の比較に時間がかかる | 物件、競合、収益を同じ画面導線で比較できる |
| 不動産会社・エージェント | 民泊向け物件を効率よく提案したい | 掲載、問い合わせ、提案材料を管理できる |
| 運営代行・法人 | 複数エリアの競合や許認可を調べたい | エリア別データと通知で候補を継続監視できる |

## 3. 現在の到達点

実装は Phase 1 の静的サイトを超えて、以下を含む。

- 住所・URL入力型の可否チェッカー
- 物件検索 `/properties`
- エリア検索 `/search`
- 物件詳細 `/property/[id]`
- 届出住宅マップ `/map`
- エリア SEO ページ `/area`
- ブログ `/blog`
- 料金ページ `/pricing`
- 問い合わせ・リード API
- 物件掲載フォーム
- Prisma/Supabase を想定した DB 層
- DB 未設定時のモックデータフォールバック

## 4. MVP 本番要件

### 4.1 可否チェック

必須要件:

- 住所または物件 URL を入力できる
- 用途地域・民泊タイプ・注意点を返す
- 競合届出住宅数と収益目安を返す
- 結果をチェック履歴に保存できる

受け入れ条件:

- 主要な住所サンプルでエラーなく判定結果が表示される
- 未対応住所でも「判定できない理由」と次アクションが表示される
- `npm run build` で静的生成に失敗しない

### 4.2 物件検索

必須要件:

- `/properties` で物件一覧を表示できる
- キーワード、都道府県、間取り、賃料、民泊タイプ、タグで絞り込める
- 検索条件が URL に反映される
- DB 未設定時も `lib/data/properties.ts` で動作する

受け入れ条件:

- タグ検索の流入先は `/properties` に統一する
- `/search` はエリア検索として扱い、物件タグ検索と混ぜない

### 4.3 競合マップ

必須要件:

- `public/data/minpaku_listings.json` を読み込む
- sourceId ごとに分かりやすい地域名を表示する
- ピンまたはクラスタで地図表示できる
- `/listings` は `/map` に統一する

受け入れ条件:

- 生の sourceId がユーザー向けラベルとして表示されない
- JSON が欠損しても壊れ方が制御されている

### 4.4 収益化

必須要件:

- Free / Standard / Pro の差分が明確である
- Stripe 未設定時に壊れたボタンに見えない
- Stripe 設定済みの場合は Checkout に遷移できる
- Webhook でプラン状態を反映する導線を持つ

受け入れ条件:

- 環境変数不足時は画面内で「準備中」と理由を表示する
- ユーザーがクリック後の alert だけで状態を知る作りにしない

### 4.5 問い合わせ・リード

必須要件:

- `/contact` から問い合わせを送信できる
- チェック結果画面でメール獲得できる
- Resend 未設定時は保存/ログなどの代替挙動を持つ

受け入れ条件:

- API は必須フィールドを検証する
- ユーザーには成功/失敗が明確に表示される

## 5. 非機能要件

品質:

- `npm run lint`
- `npx tsc --noEmit`
- `npm run build`

パフォーマンス:

- 地図はクライアント側で遅延読み込みする
- 大量ピンはクラスタリングまたは集計表示を使う

運用:

- DB 未設定でもローカル開発ができる
- Windows の Prisma DLL ロック問題を README/AGENTS に明記する
- 環境変数不足時の UI を用意する

SEO:

- `/area`, `/blog`, `/property` の metadata と JSON-LD を維持する
- sitemap/robots の対象 URL を実装ルートと同期する

## 6. 優先順位

### P0: 本番で壊れて見える箇所をなくす

- Stripe 未設定時の料金ページ UI を改善する
- Resend 未設定時の問い合わせ API 挙動を確認する
- JSON/DB 欠損時の競合マップのフォールバックを確認する

### P1: 導線と情報設計を仕上げる

- `/properties` は物件検索、`/search` はエリア検索として文言を統一する
- モバイルナビとハンバーガーの役割を整理する
- トップページのセクション順序を「誰向け → なぜ便利 → 使い方」に寄せる

### P2: データの本番化

- Supabase プロジェクトを設定する
- Prisma migration と seed を実行する
- 公開物件データの取り込み方針を確定する
- 用途地域 GIS による住所単位判定の精度を上げる

### P3: 有料機能の実体化

- プラン状態を Supabase/Prisma に永続化する
- お気に入り上限、通知、掲載上限をサーバー側でも検証する
- Stripe webhook で `Subscription` を更新する
